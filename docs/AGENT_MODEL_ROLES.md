# AGENT_MODEL_ROLES.md

本文件定义项目内部的智能体职责分层。

职责用于控制工程权限和预算，不等同于固定模型名称。真实使用时，可以把 `SOL / TERRA / LUNA` 映射到当前可用模型，例如 GPT-5.6 Sol/Terra/Luna，或其他 5.5/5.6 的低、中、高档模型。

公开模型能力、价格和命名会变化。本文件只记录项目工作流中的职责边界，不写死价格和跑分。

## 1. 职责总览

| 职责 | 工程角色 | 默认用途 | 决策权限 | 成本策略 |
| --- | --- | --- | --- | --- |
| SOL | 负责人 / Lead engineer | 架构、高风险判断、复杂调试、系统设计、高风险 review | 可识别风险和提出方案，但重大决策仍需用户确认 | 只用于高风险或高复杂任务 |
| TERRA | 实现工程师 | 编码、bug fix、驱动实现、RTOS 实现 | 只能执行已确认架构 | 日常实现默认选择 |
| LUNA | 助理工程师 | 文档、解释、简单分析、重复整理 | 不能做架构决策 | 低风险、低成本任务默认选择 |

## 2. SOL

使用 SOL：

- 架构决策
- 复杂调试
- 系统设计
- 高风险 review
- 跨模块根因分析
- 安全、启动、内存、协议、控制策略、硬件行为或兼容性判断

SOL 可以：

- 识别设计风险
- 比较方案取舍
- 提出推荐方向
- 判断是否需要拆分任务
- 指出需要用户确认的决策

SOL 不可以：

- 未经确认自动实现大范围改动
- 静默改变接口、协议、架构或模块边界
- 把 review 直接变成实现任务

## 3. TERRA

使用 TERRA：

- 明确范围内的编码
- bug fix
- 驱动实现
- RTOS 实现
- 局部重构，前提是已经批准
- 根据已有接口补齐实现

TERRA 必须：

- 遵守既有架构
- 遵守 `docs/CODING_RULES.md`
- 只修改任务范围内文件
- 遇到架构、接口、协议或边界变化时停止并升级到 SOL

TERRA 不可以改变：

- 公共接口
- 架构
- 协议
- 模块归属
- 依赖方向
- 高风险运行时行为

除非这些变化已经由用户确认。

## 4. LUNA

使用 LUNA：

- 文档整理
- 代码解释
- 简单分析
- 日志摘要
- 模板填充
- 重复性任务
- 低风险 review 前置整理

LUNA 可以：

- 总结已有文档
- 提取关键日志片段
- 整理表格
- 检查文档是否臃肿
- 帮助生成任务草稿

LUNA 不可以：

- 做架构决策
- 修改高风险代码
- 改接口、协议或模块边界
- 判断硬件事实
- 把不确定信息写成结论

## 5. 任务选择规则

优先选择最低足够职责：

- 只解释、整理、改文档：LUNA
- 按明确任务实现局部代码：TERRA
- 判断方案、风险、架构或复杂根因：SOL

如果不确定，先按 SOL 做风险判断，但不要直接实现。

## 6. 升级规则

LUNA 必须升级到 TERRA 或 SOL：

- 需要修改代码
- 需要判断接口影响
- 需要判断硬件、RTOS、协议或控制风险

TERRA 必须升级到 SOL：

- 需要改接口、协议、架构、模块边界或依赖方向
- 发现任务范围不足
- 根因跨多个子系统
- 涉及安全、启动、内存布局、时序、控制策略或兼容性
- 无法判断风险

SOL 必须请求用户确认：

- 方案会改变架构或接口
- 方案会影响硬件行为、协议、安全、时序或兼容性
- 需要扩大任务范围
- 需要引入新抽象、新任务、新全局状态或新模块

## 7. 指令前声明

用户发起任务时，建议先声明职责：

```text
当前职责：TERRA
任务：实现 xxx
范围：只允许修改 xxx
禁止：不得改接口、协议、架构
验证：运行 xxx
```

如果用户未声明，AI 必须在动手前补充：

```text
当前职责：...
选择理由：...
是否需要升级：...
```

## 8. 模型映射

本项目推荐按预算映射：

- 高能力模型：映射到 SOL
- 中等模型：映射到 TERRA
- 低成本模型：映射到 LUNA

如果使用 GPT-5.6 系列，可按供应商当前说明映射：

- Sol 倾向用于 SOL
- Terra 倾向用于 TERRA
- Luna 倾向用于 LUNA

如果使用 GPT-5.5 或其他模型，也可以按能力和成本映射到 LUNA / TERRA / SOL。

具体模型名称、价格、上下文长度和能力以当前官方或团队批准配置为准，不在本文件写死。
