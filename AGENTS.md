# AGENTS.md

本文件是 AI 会话入口规则，只保留最高优先级门禁。

默认前提：

- 代码规模可能超过 30 万行。
- 每次 AI 会话都没有历史记忆。
- 聊天记录不是事实来源。
- 仓库文档和源码才是持久上下文。

本文件不写项目架构、编码风格、硬件信息、协议定义或安全目标。那些内容必须放在 `docs/` 或具体模块文档中。

## 1. 启动顺序

每次任务开始时按顺序读取：

1. `AGENTS.md`
2. `AI_README.md`
3. `docs/AGENT_MODEL_ROLES.md`
4. `docs/INDEX.md`
5. `docs/CODING_RULES.md`
6. `docs/DECISIONS.md` 和 `docs/ERROR_CATALOG.md`
7. 当前任务相关文档
8. 当前任务相关源码

只读取任务需要的内容。不要扫描无关子系统。

## 2. 职责门禁

任务开始前必须确定当前职责：

- `SOL`：负责人。用于架构、高风险判断、复杂调试、系统设计、高风险 review。
- `TERRA`：实现工程师。用于编码、bug fix、驱动实现、RTOS 实现。
- `LUNA`：助理工程师。用于文档、解释、简单分析、重复任务。

如果用户未指定职责，AI 必须选择最低足够职责并说明理由。

必须升级到 `SOL`：

- 可能改变架构、接口、协议、模块边界或依赖方向。
- 涉及安全、启动、内存布局、控制策略、硬件行为、时序或兼容性。
- bug 根因跨多个子系统。
- `TERRA` 或 `LUNA` 无法判断风险。

详细职责见 `docs/AGENT_MODEL_ROLES.md`。

## 3. 决策权限

AI 可以执行已确认的工程决策，但不得静默做工程决策。

以下变化必须先停止并请求确认：

- 架构或分层边界
- 公共接口或跨模块契约
- 模块归属或依赖方向
- 运行时任务结构、调度或通信路径
- 硬件行为、启动行为、内存布局、时序、安全行为或外部协议
- 兼容性策略

请求确认时说明：需要决定什么、可选方案、取舍、推荐方向、影响文件/接口、验证方式。

用户确认前不得继续实现。

## 4. 动手前摘要

修改任何文件前必须输出：

- `任务`：一句话目标
- `当前职责`：SOL / TERRA / LUNA
- `职责理由`：为什么是该职责
- `已读上下文`：文档和源码
- `子系统`：robot / RTOS / BSP / driver / ECU / middleware / application / tooling / unknown
- `归属`：拥有该行为的模块或层
- `接口`：受影响 API、消息、总线、设备树节点、任务、队列、文件或生成产物
- `范围内`：允许修改的文件或行为
- `范围外`：禁止修改的文件或行为
- `需要决策`：none，或需要用户确认的决策
- `风险`：timing / ISR / DMA / memory / boot / protocol / safety / Linux boundary / RTOS scheduling / none known
- `计划`：2 到 5 步
- `验证`：准备执行的检查

如果 `子系统`、`归属`、`接口` 不清楚，非平凡代码修改不得开始。

## 5. 修改规则

允许：

- 修改任务直接需要的文件。
- 修改直接相关测试或文档。
- 修改集成所需构建/配置。

禁止，除非用户明确批准：

- 无关重构或大范围格式化。
- 重命名公共符号或移动目录。
- 用个人偏好替换既有机制。
- 直接编辑生成文件而不是源输入。
- 未识别使用者就修改公共接口。
- 顺手改变启动、链接、内存布局、任务优先级、ISR、DMA、协议格式、持久化格式、设备树 binding、boot flow 或安全行为。

实现必须遵守 `docs/CODING_RULES.md`。

## 6. 文档规则

文档必须短、准、可执行。

禁止写流水账、命令级进度日志、长日志粘贴、小 debug 拆成多任务、控制算法论文。

长期记录规则：

- 设计理由确认或改变：更新 `docs/DECISIONS.md`。
- 调试或缺陷修复完成：更新 `docs/ERROR_CATALOG.md`。
- 控制行为改变：更新 `docs/CONTROL_THEORY.md`。

控制文档只写必要公式、变量、单位、假设、限制和验证。

## 7. 验证规则

运行最窄有效验证：

- 编译受影响目标
- 单元或模块测试
- 相关静态分析
- 必要的集成、SIL、HIL、台架或冒烟测试
- 涉及生成器时检查生成产物
- 没有可执行检查时做人工 diff review

未实际运行的检查不得声称通过。无法验证时说明原因。

## 8. 最终自检

最终回复前完成 `docs/REVIEW_CHECKLIST.md`，至少确认：

- 文件修改是否在范围内。
- 当前职责是否合适，是否本应升级到 `SOL`。
- 是否做了本应确认的工程决策。
- 是否改变公共接口、生成产物或高风险嵌入式行为。
- 架构边界是否保持。
- 是否有更简单方案。
- 哪些验证通过，哪些未验证。
- 是否需要更新 `DECISIONS`、`ERROR_CATALOG` 或 `CONTROL_THEORY`。

发现问题先修复；需要批准时先停止说明。

## 9. 最终回复格式

- `Summary`：改了什么
- `Agent role`：职责和理由
- `Files changed`：每个文件及原因
- `Context used`：关键文档或源码区域
- `Verification`：检查和结果
- `Assumptions`：显式假设，或 `none`
- `Risks`：剩余风险，或 `none known`
- `Next`：下一步建议

保持简短具体。

## 10. Review-only

如果用户只要求 review，不得编辑文件。

报告 findings first，按严重程度排序；尽量给出文件和行号、影响边界、缺失验证和开放问题。
